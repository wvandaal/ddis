## 1. Executive Summary

The Braid specification and implementation guide represent a **remarkably ambitious and architecturally sound** design for a CRDT-based append-only datom store with harvest/seed lifecycle. The foundational abstractions (datom algebra, CRDT merge, content-addressable identity) are mathematically grounded. The three-level refinement methodology (L0 algebraic → L1 state machine → L2 implementation contract) is consistently applied and produces genuinely falsifiable invariants.

**However, the documents are not yet implementation-ready.** The audit identified significant divergences between the spec and guide that would cause an implementing agent to make contradictory design choices depending on which document it consulted. Five critical behavioral mismatches exist that would produce semantically different systems depending on resolution. The type catalog has 21 phantom types (referenced but never defined) and 13 types with incompatible definitions between spec and guide.

**Verdict: CONDITIONAL PASS — requires targeted remediation before Stage 0 implementation begins.**

## 3. Critical Findings (Deduplicated)

After cross-referencing all 14 agents, the 30 raw CRITICAL findings reduce to **18 unique critical issues** in 5 categories:

### Category A: Behavioral Mismatches (would produce different systems)

**A1. LWW Tie-Breaking Rule Contradiction**
- **Spec** (INV-RESOLUTION-005): "Ties broken by agent ID"
- **Guide** (04-resolution.md): "Break ties by datom hash (BLAKE3, ADR-STORE-013)"
- **Impact**: Same data produces different query results depending on which rule the implementer follows. Every resolution-bearing query depends on this.
- **Sources**: Agent 3, Agent 10, Agent 12 (#24)
- **Fix**: Single ADR deciding one rule. Recommendation: BLAKE3 hash (deterministic, agent-independent).

**A2. INV-MERGE-008 Dual Semantics**
- **Spec** (07-merge.md): "At-Least-Once Idempotent Delivery"
- **Guide** (07-merge-basic.md): "Merge receipt records operation counts"
- **Impact**: Same invariant ID maps to completely different properties. Breaks traceability graph.
- **Sources**: Agent 3, Agent 12 (#42)
- **Fix**: Renumber one. Spec's delivery semantics should keep INV-MERGE-008; guide's receipt recording needs a new ID.

**A3. MCPServer Architectural Model**
- **Spec** (ADR-INTERFACE-004): "Library model — MCP server holds &Store reference, no disk IO"
- **Guide** (09-interface.md): `MCPServer { store_path: PathBuf }` — file-based model
- **Impact**: Fundamental architectural disagreement about how the MCP server interacts with the store.
- **Sources**: Agent 6, Agent 12 (#50)
- **Fix**: Amend ADR-INTERFACE-004 with rationale for file-based model, or update guide.

### Category B: Type System Incoherence (would cause compile errors)

**B1. QueryExpr — Completely Different Types**
- **Spec**: `QueryExpr { Find { variables, clauses }, Pull { pattern, entity } }` (2-variant enum)
- **Guide**: `ParsedQuery { find_spec, where_clauses, rules, inputs }` (flat struct)
- **Impact**: Every command accepting a query depends on this type.
- **Sources**: Agent 2, Agent 8, Agent 12 (#14)
- **Fix**: Adopt guide's flat struct structure with spec's `QueryExpr` name.

**B2. SeedOutput vs AssembledContext**
- **Spec**: `AssembledContext { sections: Vec<ContextSection>, total_tokens, budget_remaining }`
- **Guide**: `SeedOutput { orientation, decisions, context, warnings, task }`
- **Guide/architecture**: Defines BOTH types, contradicting itself
- **Impact**: Seed assembly pipeline can't compile when two incompatible types exist.
- **Sources**: Agent 4, Agent 8, Agent 12 (#33, #57)
- **Fix**: Adopt `SeedOutput` (guide's five-field struct). Remove `AssembledContext` or make it internal.

**B3. Value Enum Variant Count (13 vs 9)**
- **Spec**: 13 variants (String through Json)
- **Guide**: 9 variants (defers URI, BigInt, BigDec, Tuple, Json to post-Stage 0)
- **Impact**: Cascades to ValueType, Schema validation, and every type touching Value.
- **Sources**: Agent 1, Agent 8, Agent 12 (#1, #12)
- **Fix**: Add explicit `// Stage 0` and `// Stage 1+` tags to spec. docs/guide/00-architecture.md already does this; propagate to docs/guide/01-store.md.

**B4. Transaction\<S\> Field Mismatch**
- **Spec**: `tx_data: Vec<Datom>, metadata: TxMetadata`
- **Guide/architecture**: `datoms, agent, causal_predecessors, timestamp` (different fields entirely)
- **Sources**: Agent 8
- **Fix**: Reconcile. Guide's fields are more explicit; spec's TxMetadata bundles them.

**B5. MergeReceipt — Completely Different Fields**
- **Spec**: `datoms_added, conflicts_detected, subscriptions_fired, stale_projections`
- **Guide**: `new_datoms, duplicate_datoms, frontier_delta`
- **Sources**: Agent 3, Agent 8, Agent 12 (#40)
- **Fix**: Stage-tag spec's fields. Adopt guide's Stage 0 field set.

**B6. HarvestCandidate — 7 Field Divergences**
- Multiple fields differ in name, type, or presence between spec and guide.
- **Sources**: Agent 4, Agent 8, Agent 12 (#27, #28, #29)
- **Fix**: Reconcile as a batch. Both documents need a single canonical type.

### Category C: Incomplete Formal Proofs (CRDT correctness at risk)

**C1. Semilattice Proof Incomplete**
- Partial order on P(D) never explicitly stated (reflexivity, antisymmetry, transitivity).
- Join-semilattice claim rests on implicit ⊆ ordering.
- **Sources**: Agent 3, Agent 10
- **Fix**: Add explicit partial order proof. Three lines of algebra.

**C2. Cascade Breaks Commutativity and Associativity (L1, L2)**
- `merge(A, B)` triggers cascading retractions that depend on order of evaluation.
- This means `merge(A, B) ≠ merge(B, A)` when cascades are involved.
- **Sources**: Agent 3, Agent 10
- **Fix**: Specify cascade as a post-merge deterministic fixpoint, not as part of merge itself. Then L1/L2 hold for the merge operator, and cascade is a separate commutative operation.

**C3. Causal Independence Uses Wrong Ordering Relation**
- `detect_conflicts` uses HLC ordering but HLC tracks physical time, not causal precedence.
- Two causally independent events can have ordered HLCs.
- **Sources**: Agent 3, Agent 10
- **Fix**: Use causal predecessor set, not HLC comparison, for independence detection.

**C4. User-Defined Lattice Validation Absent**
- INV-RESOLUTION-002 allows lattice-resolved attributes but no validation that user-supplied `join()` forms a valid semilattice.
- **Sources**: Agent 3, Agent 10
- **Fix**: Require semilattice witness (associativity + commutativity + idempotency tests) at schema definition time.

### Category D: Scope and Feasibility

**D1. Stage 0 Scope Unrealistic**
- 61 INVs including a Datalog query engine, graph algorithms (SCC, PageRank), guidance system, and MCP server — claimed achievable in "1-2 weeks."
- **Sources**: Agent 7, Agent 9
- **Fix**: Agent 9 showed 100% achievable with 8 simplification notes. Add these notes. Revise timeline estimate or explicitly state it assumes a capable AI agent pair.

**D2. Datalog Query Engine — Zero Implementation Guidance**
- The most complex component (semi-naive bottom-up evaluation, stratification, unification) has no worked implementation examples or pseudocode.
- **Sources**: Agent 2, Agent 7
- **Fix**: Add implementation pseudocode for semi-naive evaluation. Reference a known Rust Datalog implementation (e.g., datafrog) as structural guidance.

**D3. Kani CI Time Claim Unrealistic**
- Spec claims "< 15 minutes" for all Kani harnesses in CI.
- Reality: 34 feasible harnesses with bitvector reasoning will take 60-180 minutes.
- **Sources**: Agent 7, Agent 13
- **Fix**: Revise to "< 60 minutes with incremental caching" or split into per-namespace Kani jobs.

**D4. K_agent Harvest Detection Epistemologically Unsound**
- "Detect knowledge the agent has that isn't in the store" requires knowing what the agent knows — which is not computable from the store alone.
- **Sources**: Agent 4, Agent 5
- **Fix**: Reframe as heuristic detection (diff between session output and store content) rather than claiming formal completeness.

## 4. Systemic Patterns

Five systemic patterns account for ~60% of all findings:

### Pattern 1: Methods vs Free Functions (affects 4 namespaces)

**Spec** consistently places operations as methods on Store:
```rust
store.query(expr, mode)        // spec
store.harvest_detect(session)   // spec
store.associate(cue)            // spec
store.merge(other)              // spec
```

**Guide** consistently uses free functions:
```rust
query(store, expr, mode)              // guide
harvest_pipeline(store, session)      // guide
assemble_seed(store, task, budget)    // guide
merge(target, source)                 // guide
```

**Resolution**: One ADR should settle this project-wide. Recommendation: free functions (better modularity, clearer ownership, consistent with Rust idioms for non-self-referential operations).

### Pattern 2: Three Different Seed Section Names

| Source | Section 1 | Section 2 | Section 3 | Section 4 | Section 5 |
|--------|-----------|-----------|-----------|-----------|-----------|
| Spec ADR-SEED-004 | Orientation | Constraints | State | Warnings | Directive |
| docs/guide/06-seed.md | Orientation | Decisions | Context | Warnings | Task |
| docs/guide/00-architecture.md | Orientation | Prior Decisions | Working Context | Warnings | Task |

**Resolution**: Pick guide's naming (Orientation, Decisions, Context, Warnings, Task) — more intuitive. Update spec ADR and architecture file.

### Pattern 3: Phantom Types (21 types referenced but never defined)

Agent 8 identified 21 types used in signatures but defined nowhere:
`AssociateCue`, `ContextSection`, `TxMetadata`, `LwwClock`, `ProjectionPattern`, `SessionContext`, `ActiveSection`, `AmbientSection`, `Demonstration`, `DriftCorrection`, `ClaudeMdConfig`, `CombineStrategy`, `ComparisonCriterion`, `BranchComparison`, `SubscriptionFilter`, `BreakpointGuard`, `BreakpointCondition`, `BreakpointAction`, `DebugContext`, `DeliberationConfig`, `DecisionCriteria`

**Resolution**: Define each or remove references. Many are Stage 1+ and should be tagged accordingly.

### Pattern 4: Token Counting Undefined

Multiple agents flagged that the spec and guide reference "token count" and "token budget" without specifying a tokenizer:
- 4-character heuristic in guide (known 15-30% inaccurate)
- No tokenizer specified in spec
- Agent mode ceiling (300 tokens) depends on accurate counting
- Guidance footer ceiling (50 tokens) depends on accurate counting

**Resolution**: Specify either (a) a specific tokenizer (cl100k_base) or (b) formally define "token" as characters/4 with explicit error bounds.

### Pattern 5: Spec Internal Contradictions

Three spec-internal contradictions detected (not spec-vs-guide):
1. NEG-SCHEMA-001 contradicts ADR-SCHEMA-005 on Schema ownership model
2. Stratum monotonicity in spec/03-query.md contradicts ADRS.md SQ-002
3. INV-STORE-001 verification tag V:TYPE in matrix vs V:PROP in spec body

**Resolution**: Fix each. ADR wins over NEG. Verification matrix is authoritative for tags.

## 7. Overall Assessment

### Strengths

1. **Mathematical foundations are solid.** The datom algebra, CRDT properties, and append-only semantics are correctly grounded. The issues are in layers above the core, not in the core itself.

2. **Three-level refinement is consistently applied.** L0→L1→L2 decomposition appears throughout both spec and guide, producing genuinely falsifiable invariants. This is rare and valuable.

3. **Traceability is excellent.** 100% forward traceability from SEED.md through spec to guide. Zero contradictions between ADRS.md and spec. This is the sign of a well-organized design process.

4. **Verification is 97.8% achievable.** The spec's verification ambitions (proptest + Kani + stateright) are feasible for 182/186 INVs. The 4 infeasible items are identified and bounded.

5. **Self-bootstrap design is coherent.** All 233 spec elements are representable as datoms. Genesis bootstrapping circularity is resolved. The system can genuinely specify itself.

### Weaknesses

1. **Spec and guide were written semi-independently.** The divergence catalog (67 items) shows that the guide evolved design decisions that weren't propagated back to the spec. This is the single largest source of risk.

2. **No canonical type catalog exists.** Types are defined ad hoc in each namespace's spec and guide files. A shared `types.md` or `00-preamble.md` extension would eliminate 40+ divergences.

3. **CRDT cascade semantics are underspecified.** The most dangerous gap: cascade retractions break the two most important algebraic properties (commutativity, associativity). This is fixable but must be fixed before implementation.

4. **Stage 0 scope is ambitious.** While Agent 9 showed 100% closure is achievable with simplification notes, the raw scope (61 INVs including a Datalog engine) is aggressive. The simplification notes are load-bearing — they must be formally added.

5. **Token counting is a pervasive undefined dependency.** Multiple budget-related INVs depend on "token count" without defining a tokenizer. This creates an untestable property that cascades through BUDGET, GUIDANCE, SEED, and INTERFACE namespaces.
