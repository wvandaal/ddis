navigation\n\n* [README](#)\n* [Contributing](#)\n* [Apache-2.0 license](#)\n* [Security](#)\n\n# CXDB - AI Context Store\n\n**CXDB is an AI Context Store for agents and LLMs**, providing fast, branch-friendly storage for conversation histories and tool outputs with content-addressed deduplication.\n\nBuilt on a Turn DAG + Blob CAS architecture, CXDB gives you:\n\n* **Branch-from-any-turn**: Fork conversations at any point without copying history\n* **Fast append**: Optimized for the 99% case - appending new turns\n* **Content deduplication**: Identical payloads stored once via BLAKE3 hashing\n* **Type-safe projections**: Msgpack storage with typed JSON views for UIs\n* **Built-in UI**: React frontend with turn visualization and custom renderers\n\n## Quick Start\n\nThe fastest way to try CXDB is with the pre-built Docker image:\n\n```\n# Run the server (binary protocol :9009, HTTP :9010)\ndocker run -p 9009:9009 -p 9010:9010 -v $(pwd)/data:/data cxdb/cxdb:latest\n\n# Create a context and append a turn\ncurl -X POST http://localhost:9010/v1/contexts\n# => {\"context_id\": \"1\", \"head_turn_id\": \"0\", \"head_depth\": 0}\n\ncurl -X POST http://localhost:9010/v1/contexts/1/turns \\\n  -H \"Content-Type: application/json\" \\\n  -d '{\n    \"type_id\": \"com.example.Message\",\n    \"type_version\": 1,\n    \"payload\": {\"role\": \"user\", \"text\": \"Hello!\"}\n  }'\n\n# View in the UI\nopen http://localhost:9010\n```\n\n## Installation\n\n### From Source\n\n**Prerequisites:**\n\n* Rust 1.75+ (for the server)\n* Go 1.22+ (for directed acyclic graph (DAG):\n\n```\nTurn {\n  turn_id: u64             # Unique, monotonically increasing\n  parent_turn_id: u64      # 0 for root turns\n  depth: u32               # Distance from root\n  payload_hash: [32]byte   # BLAKE3 hash of payload\n}\n```\n\nTurns form chains:\n\n```\nroot (turn_id=1) \u2192 turn_id=2 \u2192 turn_id=3 (head)\n                     \u2193\n                   turn_id=4 \u2192 turn_id=5 (alternate branch)\n```\n\nA **Context** is a mutable branch head pointer:\n\n```\nContext {\n  context_id: u64\n  head_turn_id: u64       # Current tip of this branch\n}\n```\n\nForking is O(1): create a new context pointing to an existing turn.\n\n### Blob CAS (Content-Addressed Storage)\n\nAll turn payloads are stored in a content-addressed blob store:\n\n* Each blob is identified by `BLAKE3(payload_bytes)`\n* Identical payloads are deduplicated automatically\n* Blobs are compressed with Zstd level 3\n* Stored in `blobs/blobs.pack` with an index at `blobs/blobs.idx`\n\n### Type Registry\n\nCXDB supports typed payloads with forward-compatible schema evolution:\n\n1. **Go writers** define types with numeric field tags:\n\n   ```\n   type Message struct {\n       Role string `cxdb:\"1\"`\n       Text string `cxdb:\"2\"`\n   }\n   ```\n2. **Registry bundles** are published to the server, describing field types and names\n3. **Rust server** uses the registry to project msgpack \u2192 typed JSON for readers\n4. **React UI** consumes typed JSON with safe u64 handling and custom renderers\n\n### Renderers\n\nRenderers are JavaScript modules that visualize turn payloads:\n\n```\n// Renderer for com.example.Chart turns\nexport default function ChartRenderer({ data }) {\n  return <LineChart data={data.points} />;\n}\n```\n\nRenderers are:\n\n* Loaded from CDN (ESM modules)\n* Sandboxed with CSP\n* Hot-swappable without server restart\n\n## Architecture\n\nCXDB is a three-tier This means uncertainty propagates backward through the causal graph: if a leaf entity becomes contested, its ancestors' consequential uncertainty increases automatically.\n\nResolution mechanism: **careful staged resolution** (bilateral coordination). You can't just pick a winner \u2014 you need to assess the cascade. Resolve tentatively, compute the downstream impact, and only commit if the cascade is acceptable. This kind of uncertainty *requires simulation to assess* and is the most expensive to resolve.\n\n### The Combined Measure\n\nThe three dimensions give you a point in uncertainty space:\n\n```\nU(entity) = (\u03c3_e, \u03c3_a, \u03c3_c) \u2208 [0,1]\u00b3\n```\n\nThe scalar `uncertainty(entity)` that drives delegation is a *projection* of this vector:\n\n```\nuncertainty(entity) = ||U(entity)||_w = \u221a(\u03b1\u00b2\u03c3_e\u00b2 + \u03b2\u00b2\u03c3_a\u00b2 + \u03b3\u00b2\u03c3_c\u00b2)\n```\n\nBut the vector form is more informative than the scalar because **different dimensions require different resolution strategies**. A coordinator that knows \u03c3_e is high but \u03c3_a is low knows to dispatch more exploration agents, not more deliberation. A coordinator that knows \u03c3_c is high knows to proceed cautiously even if \u03c3_e and \u03c3_a are zero.\n\n### Why This Is the Right Formulation\n\nThe uncertainty tensor gives you something no existing system has: **a computable, decomposed measure of why something is uncertain and what to do about it.** Most distributed systems treat uncertainty as a binary (conflict / no conflict) or a scalar (conflict count). The tensor form preserves the *structure* of the uncertainty, which is exactly the information the coordination layer needs to make good decisions.\n\nIt also connects back to DDIS in a precise way. DDIS's existing 5-dimensional quality vector (coverage, depth, coherence, completeness, formality) is a measure of *spec quality*. The uncertainty tensor is a measure of *entity quality* in the fact store. The two compose: the spec fitness function F(S) can incorporate the uncertainty tensor as an additional signal:\n\n```\nF(S) = ... existing terms ... + \u03b4 \u00d7 (1 - mean_uncertainty(S))\n```\n\nWhere `mean_uncertainty(S)` is the average scalar uncertainty across all entities. This makes the bilateral loop converge toward a state where not only is the spec complete and drift-free, but the fact store itself is maximally certain.\n\n---\n\n## Your Second Question: Delegation Formulation\n\nYou asked for the single smartest addition to the delegation model.
